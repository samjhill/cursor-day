# DEMO.md — Show & Tell Playbook

## The 30-Second Script

Practice this until it's muscle memory. Time each section.

| Sec | Beat | Say / Do |
|-----|------|----------|
| 0–5 | **Hook** | "I'm [name]. I built **[Project Name]** in 2.5 hours with Cursor at Cook Day." |
| 5–10 | **Problem** | "[One sentence: who has this problem and why it sucks today.]" |
| 10–25 | **Live demo** | Show ONE interaction. Input → result. No scrolling, no settings. |
| 25–28 | **Cursor angle** | "Cursor Agent wrote most of this — I picked ingredients on my kitchen page and steered with prompts." |
| 28–30 | **Close** | "Live at [URL]" or "Repo on my GitHub." Smile. Done. |

---

## Demo Do's

- **Stand, speak up, smile** — energy wins tiebreakers
- **Big clicks, slow mouse** — audience/projector needs to follow
- **Pre-fill demo input** — copy-paste beats live typing under pressure
- **Dark mode** — already done in this scaffold ✓
- **End on the wow** — last frame should be the impressive output

---

## Demo Don'ts

- Don't apologize ("it's rough but…")
- Don't show code unless asked
- Don't explain architecture
- Don't demo login flows
- Don't say "and also I was going to add…"

---

## Judging Signals (unofficial — read the room)

Based on typical builder events + Cursor audience:

1. **Works live** — #1 factor
2. **Clear value prop** — understood in 10 seconds
3. **Visual polish** — looks intentional, not hackathon-default
4. **Cursor story** — meta narrative about how you built it
5. **Memorable** — fun, surprising, or genuinely useful
6. **Deployed** — shareable URL is a flex

---

## Backup Plans

| Failure | Recovery |
|---------|----------|
| WiFi dies | Demo from localhost + phone hotspot for judges after |
| API key fails | Switch to mock mode (already built into `/api/cook`) |
| Bug on stage | "Here's what it should do" + screenshot. Stay calm. |
| Run over time | Skip Cursor angle, keep demo + URL |
